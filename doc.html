1. Introduction à dbt

Définition

dbt (data build tool) est un framework de transformation SQL qui applique les bonnes pratiques du développement logiciel au domaine data : modularité, versionning, tests, documentation, gestion des dépendances. dbt ne déplace pas la donnée — il génère et orchestre du SQL dans le data warehouse cible (BigQuery, Snowflake, Databricks…).

Dans notre cas, dbt s'appuie sur BigQuery comme moteur d'exécution. Chaque modèle dbt est matérialisé en table ou vue BQ.

Structure d'un projet dbt avec BigQuery

dbt_project/
├── dbt_project.yml           # Config projet (nom, paths, matérialisations par défaut)
├── profiles.yml              # Config de connexion BigQuery
├── packages.yml              # Dépendances (dbt_utils, audit_helper, etc.)
├── models/                   # Les transformations SQL
│   ├── bronze/               # Couche brute / staging
│   ├── silver/               # Couche intermédiaire
│   └── gold/                 # Couche métier exposée
├── tests/
│   ├── generic/              # Tests génériques réutilisables (reconcile_aggregates)
│   └── reconciliation/       # Tests singuliers shadow (synthesis, tv)
├── macros/                   # Code Jinja/SQL réutilisable
│   ├── audit/                # log_test_results_to_monitoring
│   ├── checks/               # query_synthesis_src, query_tv_src, etc.
│   └── tests/                # reconcile_sql (générateur de SQL de comparaison)
├── snapshots/                # Captures SCD type 2
├── seeds/                    # Fichiers CSV statiques (mapping, référentiels)
└── target/                   # Artefacts générés (manifest.json, SQL compilé)

Avantages de dbt

  • Modularité — chaque transformation est un fichier SQL indépendant, réutilisable via ref() et source()
  • Lineage automatique — dbt construit le graphe de dépendances et orchestre les runs dans le bon ordre
  • Tests first-class — tests génériques (not_null, unique, relationships) + tests custom + singular tests
  • Idempotence — rejouer un run produit le même résultat ; pas de "magie" d'état
  • Versionning — tout le code transformation vit dans Git, avec code reviews et CI
  • Documentation vivante — les descriptions YAML génèrent une doc navigable avec le lineage
  • Templating Jinja — logique conditionnelle, boucles, macros pour factoriser le SQL
  • Performance — les transformations tournent dans le warehouse (push-down compute), pas en pull

2. Introduction à Cosmos

Le problème résolu

dbt gère parfaitement les transformations, mais n'orchestre pas l'ensemble d'un pipeline data (ingestion, dépendances inter-projets, re-runs ciblés, SLA). Côté Airflow, on peut appeler dbt via un BashOperator, mais on perd alors toute la granularité du DAG dbt : chaque modèle devient invisible, les retries sont "tout ou rien", impossible de cibler un seul modèle.

La solution Cosmos

Cosmos (package Astronomer open source) convertit un projet dbt en TaskGroup Airflow, où chaque modèle dbt devient une tâche Airflow distincte. On obtient :

  • Une task Airflow par modèle dbt (run + test couplés)
  • Les dépendances Airflow reflètent exactement le DAG dbt
  • Retry granulaire modèle par modèle
  • Observabilité Airflow native (logs, durées, XCom, alerting) au niveau modèle
  • Pas de duplication de config : un seul dbt_project.yml pilote les deux mondes

Modes de chargement (LoadMode)

ModePrincipeContexte recommandé
LoadMode.DBT_LS Exécute dbt ls au parsing du DAG Dev local, petits projets
LoadMode.DBT_MANIFEST Lit un manifest.json pré-généré Prod — rapide, pas besoin de dbt au scheduler
LoadMode.AUTOMATIC Essaie MANIFEST puis DBT_LS Mise en route initiale
LoadMode.CUSTOM Parser Jinja interne, ignore dbt Projets très simples uniquement

En prod, nous utilisons DBT_MANIFEST avec un manifest.json généré en CI/CD et déployé avec les DAGs. Cela évite de faire tourner dbt ls depuis le scheduler Airflow, ce qui aurait impliqué d'y installer dbt et d'y exposer les credentials BigQuery — principe du moindre privilège.

3. Intégration avec Airflow 3

Configuration du profil via Airflow Connection

Plutôt que de monter un fichier keyfile dans les workers, nous utilisons une Airflow Connection alimentée par Google Cloud Secret Manager, et Cosmos génère le profil dbt à la volée : 

from cosmos import DbtDag, ProjectConfig, ProfileConfig, RenderConfig, ExecutionConfig
from cosmos.profiles import GoogleCloudServiceAccountDictProfileMapping
from cosmos.constants import LoadMode, TestBehavior
from pathlib import Path
from datetime import datetime

DBT_PROJECT_PATH = Path("/opt/airflow/dbt/paid_media")
MANIFEST_PATH = DBT_PROJECT_PATH / "target" / "manifest.json"

profile_config = ProfileConfig(
    profile_name="paid_media",
    target_name="prod",
    profile_mapping=GoogleCloudServiceAccountDictProfileMapping(
        conn_id="google_cloud_default",
        profile_args={
            "project": "mon-gcp-project-prod",
            "dataset": "analytics",
            "location": "EU",
            "threads": 8,
        },
    ),
)

dbt_paid_media_dag = DbtDag(
    dag_id="dbt_paid_media",
    project_config=ProjectConfig(
        dbt_project_path=DBT_PROJECT_PATH,
        manifest_path=MANIFEST_PATH,
    ),
    profile_config=profile_config,
    render_config=RenderConfig(
        load_method=LoadMode.DBT_MANIFEST,
        test_behavior=TestBehavior.AFTER_EACH,
    ),
    execution_config=ExecutionConfig(
        dbt_executable_path="/opt/airflow/dbt_venv/bin/dbt",
    ),
    schedule=[raw_paid_media_asset],     # déclenché par asset (voir section 5)
    start_date=datetime(2026, 1, 1),
    catchup=False,
    tags=["dbt", "paid_media"],
)

Breaking changes Airflow 3 à noter

  • airflow.datasets.Dataset devient airflow.sdk.Asset (l'ancien alias reste disponible)
  • Les décorateurs @dag et @task sont importés depuis airflow.sdk
  • Les outlets sur les tâches émettent des "asset events" consommables par d'autres DAGs
  • La nouvelle UI expose une vue Assets dédiée pour visualiser producteurs / consommateurs

4. Migration du DAG etl_paid_media_update vers full dbt

L'existant legacy

Le DAG etl_paid_media_update orchestrait un pipeline monolithique où chaque étape était un script Python ou SQL embarqué dans un PythonOperator / BigQueryInsertJobOperator. Caractéristiques :

  • Transformations SQL dispersées dans des fichiers .sql isolés, avec templating manuel {project_id}.{dataset}.{table}
  • Aucun lineage, aucune doc auto-générée
  • Tests de réconciliation implémentés en Python, pilotés par des fichiers JSON de configuration
  • Re-run granulaire impossible : en cas d'échec, rejoue complet ou rien
  • Dépendances inter-DAGs gérées par ExternalTaskSensor avec alignement temporel fragile

Domaines couverts par le pipeline

DomaineDataset sourceCible
GCM (Google Campaign Manager)dw_paid_media_main_metrics.gcmdm_data_media.online_publishing
Radio (offline)dw_paid_media_gsheets.off_radiodm_data_media.radio
TVdw_paid_media_tv.* (multi-tables)dm_data_media.tv
Synthesis multi-canauxagrégation de 8 sourcesdm_data_media.synthesis
Wealth Management webdw_web_analytics_adobe_*dm_data_media.focus_site

Cible full dbt

Chaque transformation SQL devient un modèle dbt dans une architecture medallion :

  • Bronze : ingestion brute, typage, dé-duplication basique (matérialisation table, partitionnée)
  • Silver : nettoyage, normalisation, jointures référentielles (matérialisation view ou incremental)
  • Gold : agrégats métier exposés aux consommateurs BI (matérialisation view le plus souvent)

Gains immédiats de la migration

  • Lineage complet visible dans dbt docs et dans l'UI Airflow via Cosmos
  • Re-run ciblé : dbt run --select gold.prix_realise_elec_fr_retail+ rejoue un modèle et toute sa descendance
  • Tests unitaires au niveau modèle couplés au run
  • Portabilité dev/prod via le système de target et de vars dans profiles.yml
  • Pas de secrets en clair : credentials via Secret Manager + Airflow Connection

5. Déclenchement du DAG dbt par Assets Airflow

Principe

Au lieu d'un scheduling par cron fragile ou d'un ExternalTaskSensor coûteux en ressources, nous utilisons les Assets d'Airflow 3 pour déclencher automatiquement le DAG dbt dès que ses données amont sont prêtes.

DAG producteur (ingestion)

from airflow.sdk import dag, task, Asset
from datetime import datetime

# URI canonique BigQuery = identifiant unique de l'asset
raw_paid_media_asset = Asset(
    name="raw_paid_media",
    uri="bigquery://mon-gcp-project-raw/dw_paid_media_main_metrics/gcm",
)

@dag(
    dag_id="ingestion_paid_media",
    start_date=datetime(2026, 1, 1),
    schedule="0 6 * * *",
    catchup=False,
)
def ingestion_dag():

    @task(outlets=[raw_paid_media_asset])   # <-- PRODUIT l'asset
    def load_bronze():
        # logique d'ingestion API / GCS -> BigQuery
        ...

    load_bronze()

ingestion_dag()

DAG consommateur (dbt)

# Même asset (URI identique)
raw_paid_media_asset = Asset(
    name="raw_paid_media",
    uri="bigquery://mon-gcp-project-raw/dw_paid_media_main_metrics/gcm",
)

dbt_paid_media_dag = DbtDag(
    dag_id="dbt_paid_media",
    ...
    schedule=[raw_paid_media_asset],    # <-- CONSOMME l'asset
    ...
)

Avantages

  • Zéro sensor : plus de tâche qui "attend" une autre DAG, plus d'alignement temporel
  • Découplage : le DAG d'ingestion ne connaît pas ses consommateurs
  • Vue Assets dans l'UI Airflow : graphe producteurs/consommateurs navigable
  • Déclenchement réactif : dès que l'asset est produit, le DAG dbt démarre
  • Combinaisons : AssetAny / AssetAll permettent des conditions fines (déclencher si A ET B, ou si A OU B)

Convention adoptée : les Asset sont centralisés dans dags/assets.py pour éviter toute divergence d'URI entre producteur et consommateur, qui casserait silencieusement la chaîne.

6. Refactoring des tests de réconciliation via macros

Le legacy

Le système legacy s'appuyait sur des fichiers JSON de configuration associés à des templates SQL : 

{
  "checking_name": "gcm_data_integrity_test",
  "src": {
    "dataset": "dw_paid_media_main_metrics",
    "tbl": "gcm",
    "col_express": [
      "SUM(impressions) as total_impress",
      "SUM(clicks) as total_clicks",
      "COUNT(DISTINCT placement_id) as total_placement"
    ]
  },
  "comp": {
    "dataset": "dm_data_media",
    "tbl": "online_publishing",
    "col_express": [
      "SUM(impressions) as total_impress",
      "SUM(clicks) as total_clicks",
      "COUNT(DISTINCT placement_id) as total_placement"
    ]
  }
}

Un moteur Python générait deux DataFrames à partir des templates SQL et comparait l'égalité. L'approche marchait, mais souffrait de limites :

  • Logique de test dupliquée en dehors du code des transformations
  • Pas d'intégration avec le lineage ni avec les runs
  • Fichiers de config à maintenir manuellement, sans validation
  • Alerting implémenté ad-hoc en Python

La cible — deux patterns dbt

Après analyse, les 5 tests legacy se répartissent en deux familles nécessitant deux approches :

FamilleTests concernésMacro utilisée
Template-driven
(SELECT paramétré)		 gcm, radio, wealth_management reconcile_aggregates (generic test)
SQL-file-driven
(logique figée complexe)		 synthesis, tv reconcile_sql (singular test)

Architecture shadow

Contrainte forte : aucun artefact intermédiaire en warehouse. Les tests ne doivent pas créer de modèles "check" à maintenir. Solution : encapsuler le SQL legacy dans des macros Jinja, et les composer via des singular tests. 

dbt_project/
├── macros/
│   ├── checks/                      # SQL legacy encapsulé (aucune matérialisation)
│   │   ├── query_synthesis_src.sql  # ex-check_dm_all_channels.sql
│   │   ├── query_synthesis_comp.sql
│   │   ├── query_tv_src.sql         # ex-check_dw_paid_media_tv.sql
│   │   └── query_tv_comp.sql
│   └── tests/
│       └── reconcile_sql.sql        # Générateur de SQL de comparaison
├── tests/
│   ├── generic/
│   │   └── reconcile_aggregates.sql # Test générique paramétré
│   └── reconciliation/              # Singular tests shadow
│       ├── synthesis_data_integrity_test.sql
│       └── tv_data_integrity_test.sql
└── models/
    └── _sources.yml                 # Tests attachés aux sources

Macro reconcile_aggregates (test générique)

Utilisée pour les tests "template-driven" — la macro reçoit directement les col_express, group_by, where depuis le YAML et génère la comparaison. Gère deux modes :

  • Mode scalaire (join_keys=[]) : un CROSS JOIN simple entre deux lignes d'agrégats totaux
  • Mode row-level (join_keys=[...]) : un FULL OUTER JOIN sur les dimensions, avec détection des lignes manquantes d'un côté comme de l'autre

Exemple d'utilisation dans _sources.yml :

sources:
  - name: dw_paid_media_main_metrics
    database: "{{ var('raw_project') }}"
    schema: dw_paid_media_main_metrics
    tables:
      - name: gcm
        tests:
          - reconcile_aggregates:
              name: gcm_data_integrity_test
              compare_relation: "source('dm_data_media', 'online_publishing')"
              src_col_express:
                - "SUM(impressions) as total_impress"
                - "SUM(clicks) as total_clicks"
                - "COUNT(DISTINCT placement_id) as total_placement"
                - "COUNT(DISTINCT SPLIT(campaign,'_')[SAFE_ORDINAL(3)]) as total_cpgn"
              comp_col_express:
                - "SUM(impressions) as total_impress"
                - "SUM(clicks) as total_clicks"
                - "COUNT(DISTINCT placement_id) as total_placement"
                - "COUNT(DISTINCT campaign_naming) as total_cpgn"
              tolerance_pct: 0.0
              config:
                store_failures: true
                schema: dbt_reconciliation
                severity: warn
                tags: ['reconciliation']

Macro reconcile_sql (générateur pour singular tests)

Utilisée pour les tests "SQL-file-driven" où la logique est trop complexe pour tenir dans un YAML. Le SQL legacy est encapsulé dans une macro query_*.sql et la comparaison est tissée au moment du test. 

-- tests/reconciliation/synthesis_data_integrity_test.sql
{{ config(
    tags=['reconciliation', 'synthesis'],
    severity='warn',
    store_failures=true,
    schema='dbt_reconciliation'
) }}

{{ reconcile_sql(
    src_sql         = query_synthesis_src(),
    comp_sql        = query_synthesis_comp(),
    compare_columns = [
        'impressions', 'clicks', 'cost', 'vv100',
        'insertion_contact_grp_cinema', 'circulation_or_spot', 'tech_fees'
    ],
    tolerance_pct   = 0.0
) }}

Au runtime, dbt compile ce test en une unique requête qui inline les deux SQL comme CTEs et calcule les écarts. Aucune vue n'est créée, rien n'est persisté — sauf si le test échoue, auquel cas store_failures: true stocke les lignes divergentes dans dbt_reconciliation.<test_name> pour analyse.

Gestion des divergences de structure

Point clé : les tables legacy et cible ont des structures différentes (colonnes renommées, casts de dates, filtres métier). La macro prend cela en charge via :

  • src_col_express / comp_col_express : expressions SQL distinctes des deux côtés
  • src_where / comp_where : filtres WHERE indépendants
  • src_group_by / comp_group_by : group by indépendants
  • Alias de colonnes identiques à la sortie, garantissant la correspondance pour la comparaison

Exemple concret — test wealth management, qui casse un format de date en chaîne côté legacy :

- reconcile_aggregates:
    name: wealth_management_data_integrity_test
    compare_relation: "source('dm_data_media', 'focus_site')"
    src_col_express:
      - "CAST(FORMAT_DATE('%Y-%m-%d', PARSE_DATE('%B %d, %Y', date)) AS DATE) as date"
      - "SUM(visits) as total_visits"
      - "SUM(page_views) as total_page_views"
    comp_col_express:
      - "date"
      - "SUM(visits) as total_visits"
      - "SUM(page_views) as total_page_views"
    src_group_by: "date"
    comp_group_by: "date"
    join_keys: ['date']
    comp_where: "source = 'adobe_wealth_management'"
    tolerance_pct: 0.0

7. Logging centralisé des exécutions de tests

Objectif

Au-delà du comportement immédiat (échec = DAG Airflow en erreur), nous avons besoin d'un journal historique de toutes les exécutions de tests pour :

  • Monitoring long terme de la qualité (dashboard Looker)
  • Calcul de SLA / SLO sur les tests critiques
  • Détection des tests instables (flapping)
  • Audit réglementaire

Mécanisme : on-run-end hook

Une macro log_test_results_to_monitoring est invoquée automatiquement à la fin de chaque dbt test / dbt build via un hook on-run-end dans dbt_project.yml : 

on-run-end:
  - "{{ log_test_results_to_monitoring() }}"

La macro exploite les variables globales exposées par dbt dans le scope du hook : results (statuts de chaque nœud exécuté), invocation_id (UUID du run), run_started_at (horodatage).

Table de journal BigQuery

CREATE TABLE IF NOT EXISTS `mon-projet.dbt_audit.dbt_test_results` (
    test_execution_id         STRING,
    test_execution_timestamp  TIMESTAMP,
    test_name                 STRING,
    test_type                 STRING,      -- not_null / unique / reconcile_aggregates / custom
    attached_node             STRING,      -- model ou source testé
    attached_node_type        STRING,      -- 'model' ou 'source'
    column_name               STRING,
    test_status               STRING,      -- pass / fail / warn / error
    failures                  INT64,
    execution_time_s          FLOAT64,
    severity                  STRING,
    target_name               STRING,
    message                   STRING,
    tags                      ARRAY<STRING>
)
PARTITION BY DATE(test_execution_timestamp)
CLUSTER BY test_status, test_name

Exemples de requêtes d'exploitation

Tests en échec sur les 7 derniers jours :

SELECT test_execution_timestamp, test_name, failures, message
FROM `mon-projet.dbt_audit.dbt_test_results`
WHERE test_status IN ('fail', 'error', 'warn')
  AND test_execution_timestamp >= CURRENT_TIMESTAMP() - INTERVAL 7 DAY
ORDER BY test_execution_timestamp DESC;

Taux d'échec par test sur 30 jours (détection des tests instables) :

SELECT
    test_name,
    COUNT(*) AS nb_runs,
    COUNTIF(test_status IN ('fail', 'error')) AS nb_failures,
    ROUND(100 * SAFE_DIVIDE(COUNTIF(test_status IN ('fail', 'error')), COUNT(*)), 1) AS fail_rate_pct
FROM `mon-projet.dbt_audit.dbt_test_results`
WHERE test_execution_timestamp >= CURRENT_TIMESTAMP() - INTERVAL 30 DAY
GROUP BY test_name
HAVING fail_rate_pct > 0
ORDER BY fail_rate_pct DESC;

8. Bilan : avantages de la refacto globale

Granularité retry Rejouer tout ou rien Retry par modèle, re-run ciblé via --select
Lineage Inexistant, documenté sur Confluence manuellement Auto-généré, navigable (dbt docs + UI Airflow)

Réconciliation Comparaison DataFrame, pas d'historique Tests shadow, journal BQ, dashboard Looker
Dépendances inter-DAG ExternalTaskSensor + alignement cron Airflow Assets, déclenchement événementiel

Code review Scripts éparpillés, revues difficiles Modèles SQL versionnés, schéma en YAML, CI dbt
Documentation Désynchronisée du code Inline dans YAML, générée par dbt docs
Parsing DAG Airflow N/A manifest.json pré-compilé, zéro dbt au scheduler

Bénéfices qualitatifs

  • Maintenabilité — un SQL dbt est plus lisible qu'un PythonOperator qui exécute une f-string SQL ; les modifications font l'objet de PRs claires
  • Onboarding — un data engineer connaissant dbt peut contribuer sans apprendre le code maison ; les conventions sont standardisées
  • Fiabilité — l'idempotence native de dbt + les retries granulaires de Cosmos réduisent les interventions manuelles
  • Observabilité — qualité des données journalisée, exposable sur dashboard
  • Découplage — les DAGs ne sont plus couplés par timing mais par événements (Assets)
  • Testabilité — les tests de réconciliation sont capitalisés et réutilisables via macros, au lieu d'être dupliqués en config JSON

Points d'attention pour la suite

  • Stabilisation des tolérances : les tests sont pour l'instant en severity: warn pendant la phase de migration. Ils passeront en severity: error une fois les écarts résiduels résorbés (notamment la partie FOURNITURE identifiée sur l'électricité retail).
  • Génération du manifest en CI : la génération du manifest.json à chaque PR mergée doit être fiabilisée ; un manifest obsolète provoque des DAGs Airflow incohérents avec le code dbt.
  • Alerting Slack : branchement prévu sur la table dbt_audit.dbt_test_results pour notifier l'équipe data lorsqu'un test critique échoue deux fois de suite.
  • Extension à d'autres pipelines : l'architecture (Cosmos + Assets + macros de réconciliation) est transposable aux autres domaines legacy, qui feront l'objet de migrations successives.
